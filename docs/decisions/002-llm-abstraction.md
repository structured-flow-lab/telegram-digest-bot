# ADR 002 — LLM-абстракция: Protocol + factory

## Статус
Принято

## Контекст

В Personal MVP используется один провайдер — Claude Haiku. Но уже сейчас известно,
что в multi-user MVP потребуется:
- несколько провайдеров (Gemini для бесплатного tier, Claude/OpenAI для платного);
- выбор модели на уровне тарифного плана пользователя;
- возможность сменить провайдера без переписывания бизнес-логики.

Если вызов Claude hardcode-ить прямо в `summarizer.py`, при любом расширении придётся
трогать digest-логику — нарушение принципа разделения ответственности.

## Решение

Ввести тонкий слой `app/llm/` из трёх файлов:

```
app/llm/
  base.py      — LLMResult (dataclass) + LLMClient (Protocol)
  claude.py    — ClaudeClient реализует LLMClient
  factory.py   — get_llm_client() читает LLM_PROVIDER из конфига
```

`summarizer.py` работает только с `LLMClient` Protocol — не знает о конкретной модели.

Конфиг:
```
LLM_PROVIDER=claude
LLM_MODEL=claude-haiku-4-5-20251001
```

Добавление нового провайдера = новый файл `app/llm/gemini.py` + одна строка в `factory.py`.

## Последствия

**Плюсы:**
- `summarizer.py` не меняется при смене провайдера.
- Легко тестировать: заменить `ClaudeClient` на mock-реализацию Protocol.
- Multi-user MVP: выбор модели по тарифу — одна строка в `factory.py`.

**Минусы:**
- Небольшой оверхед для Personal MVP (три файла вместо одного прямого вызова).
- Нужно следить, чтобы все клиенты реализовывали Protocol полностью.

## Альтернативы

- **Прямой вызов в summarizer.py** — проще сейчас, но потребует рефакторинга при
  добавлении второго провайдера. Отклонено.
- **LangChain / LiteLLM** — готовая абстракция, но лишняя зависимость для простой
  задачи с одним промптом. Отклонено до появления реальной потребности.
